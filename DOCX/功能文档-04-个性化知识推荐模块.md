# 功能文档 - 模块四：个性化知识推荐模块

本文档描述**个性化知识推荐模块**的目标能力、数据流与排序逻辑，供需求对齐、论文表述与后续迭代参考。

**说明**：文中配置项名、接口参数以工程 `application.yml` 与 Controller 为准，会随优化调整；**设计原则**（行为日志 → 用户画像 → 候选过滤 → 多因子打分 → 可选 AI 微调 → 分页与解释）应保持稳定。知识内容的组织维度以**分类**为主，本模块画像与排序信号不依赖「内容标签」类附加维度。

---

## 一、模块目标与边界

| 项目 | 说明 |
|------|------|
| 核心能力 | 在「已上架消防知识」范围内，结合用户行为与内容特征生成**有序列表**；可选调用大模型对**列表前段**重排并为**当前页**生成短理由。 |
| 典型入口 | 用户端「智能推荐」列表接口（分页、可选关键词、可选 AI）。按时间/全量浏览通常走**内容列表接口**，与本模块解耦。 |
| 非目标 | 不替代搜索引擎；不保证实时强一致（行为写入与列表可见之间允许短延迟）；不承诺单一因子（如仅协同过滤）独立最优。 |

---

## 二、行为数据与用户画像

### 2.1 行为类型与对象

- **行为类型**（示例）：浏览、点赞、评论、收藏等，入库为规范化枚举（如 VIEW / LIKE / COMMENT / COLLECT）。
- **对象类型**（示例）：知识正文、论坛帖、评论、新闻等，用 `target_type` + `target_id` 区分。
- **约束**：`target_id` 须为正整数；非法请求应在接入层拒绝并返回明确业务错误，避免污染日志表。

行为既可由 **`POST /recommend/behavior`** 显式上报，也可在业务链路内（详情浏览、收藏、论坛互动等）**统一写入**同一张行为表，保证推荐与统计同源。

### 2.2 兴趣画像（分类维度）

- 从用户对**知识类内容**的历史行为中，按**知识分类**聚合偏好；宜采用**行为类型权重**与**时间衰减**（近期行为权重大于远期），得到有序的分类偏好列表（如 Top3）。
- **冷启动**：无足够行为时，可使用运营配置的**默认兴趣分类**。
- **未登录**：仅冷启动与全站热度/时效，**不使用**个人行为画像、Item-CF、语义与论坛加成；**AI 重排与推荐理由**仅在登录且满足 6.2 条件时执行。
- **负向偏好**：用户标记「不感兴趣」的分类应与运营侧「排除分类」**合并**，在候选阶段统一过滤。

画像输出用于两件事：**规则打分**中的分类匹配加分；**AI 上下文**中的「偏好分类名称」等可读信号（通常取偏好序列的首项展示名即可）。

## 三、候选集构建（召回层）

在精排打分之前，需确定**参与打分的知识条目集合**。推荐采用与业界一致的**两阶段**思路：**召回（Retrieval）**缩小规模，**排序（Ranking）**算分。

### 3.1 多路召回（默认开启，`recommend.recall.*`）

在**无全表扫描**的前提下合并多路候选（均尊重关键词、上架状态与分类排除）：

| 通道 | 含义 |
|------|------|
| **兴趣类目** | 用户偏好 Top3（含冷启动默认类目）下，按**创建时间倒序**各取若干条，覆盖「近期偏好领域」内容。 |
| **Item-CF** | 将用户历史行为算出的协同过滤得分排序，取 Top-N **内容 id** 并入池（与精排中的 CF 加分同源信号）。 |
| **热度暖池** | 按浏览/点赞倒序分批补足，直至达到 `max-candidates` 上限，承担**冷启动与长尾覆盖**。 |

合并后对 id **去重保序**，再加载实体进入精排。若召回结果为空则**回退**为旧版「全量 id 升序拉取」（兼容小库与极端过滤）。

`total` 在启用召回截断时表示**本序参与精排的条数**；全库符合条件的条数可能更大，**pageHint** 中会说明。

### 3.2 全量/二次截断

- **`recommend.smart-max-pool`**：在召回池之上再按热度截断（0 表示不截）。
- **状态 / 关键词 / 分类过滤**：与旧版一致；注意 **NOT IN** 与 **NULL 分类** 的三值逻辑。

## 四、规则排序模型（精排层，多因子融合）

对**召回池**中每条知识计算**标量得分**，按得分降序排序；同分宜有稳定次序（如按内容 ID）。

热度 log 权重、时效衰减上限、类目阶梯加分、CF 分项 cap 等可通过 **`recommend.ranking.*`** 配置，便于毕设中对照论文公式 \(Score \approx W_{cat}\cdot match + W_{hot}\cdot \log(views) + W_{time}\cdot decay\) 做消融说明。

| 因子 | 含义 | 要点 |
|------|------|------|
| **偏好分类** | 内容与用户 Top-K 偏好分类的匹配程度 | Top1 / Top2 / Top3 分级加分（可配置数值）。 |
| **Item-CF（基于物品的协同过滤）** | 与用户曾互动过的知识「被同一用户群体共同互动」的其它知识 | 共现强度带行为权重与时间衰减；精排中对 CF 原始分设 **cap+倍数** 防止单因子碾压。 |
| **热度与互动** | 浏览量、点赞量等 | **对数缩放**（权重可配）避免头部垄断。 |
| **时效** | 发布时间越远略减分 | 指数衰减，**半衰期天数**可配（`decay-half-life-days`）。 |
| **语义相似（轻量）** | 用户近期浏览过的知识文本与候选标题/摘要的相似度 | Dice 二元组等；限制参考条数与加分上限。 |
| **跨域信号（可选）** | 论坛近期互动标题与候选知识的文本相似度 | 与知识行为互补，权重通常低于主域信号。 |

**Item-CF 在工程中的典型形态**：对每个用户近期触达过的若干内容 id，查询「与其共现的其它内容 id」及强度，在内存中按候选 id **累加**得到协同分，再写入总分的加法项——**不**必单独暴露「纯 CF 列表」接口，除非产品需要。

## 五、分页与列表响应

- 排序在**完整候选序**（或截断池序）上进行，再按 **pageNum / pageSize** 切片。
- 响应宜包含：**列表数据**、**总条数**、**分页元数据**、**策略说明文案**（如是否启用 AI、是否截断池、是否未登录冷启动等），便于前端展示与课堂演示。
- 列表项除标题、摘要、封面、统计字段外，可附带 **推荐来源标识**、**排序策略简述** 等元数据，用于可解释性展示；宜返回 **`updateTime`（内容更新时间）**，便于前端或缓存层将「候选内容版本」纳入推荐理由等缓存键，避免内容已更新仍命中旧文案。

## 六、可选：AI 增强层

### 6.1 定位

AI **不替代**规则候选与主排序，而是在规则结果之上提供：

1. **头部重排**：仅对排序后的**前 N 条**（N 可配置）调用模型或通用推荐服务，重新排列这 N 条的顺序；**第 N+1 条及以后**保持规则序不变，以控制成本与延迟。
2. **当页理由**：对**当前页**每条（或每条可选）生成短 **recommendReason**。

### 6.2 启用条件（逻辑上应同时满足）

- **请求用户已登录**（有稳定 `userId`）；未登录即使 `useAi=true` 也不调用大模型。
- 客户端或产品开关请求 AI（如 `useAi=true`）。
- 服务端 AI 总开关开启，且 URL / 密钥 / 模型等**就绪**（豆包类需完整 chat 配置；通用 HTTP 模式至少需可达的网关地址）。
- 可对 AI 列表接口单独限流（按 IP、按用户等），防止刷爆模型配额。

任一条件不满足或调用失败时，应**静默降级**：列表仍为规则序，理由可为空，并在 **pageHint** 中说明原因（可选）。

### 6.3 输入信号（供 Prompt 或通用 JSON）

- **任务指令**：固定领域（消防科普）、任务（排序 / 生成理由）、输出格式（如 id 逗号分隔、每行一条理由）。
- **用户侧**：检索词（若有）、偏好分类名称（来自画像首项或默认）、运营热点词列表（可选）。
- **条目侧**：每条候选的 id、标题、摘要（宜截断）、**分类名**。

通用 HTTP 契约可设计为：`userId`、`candidates`、`keywords`、`context`（如 `topCategoryName`、`keyword`、`hotTopics`），响应支持 `order` / `scores`（重排）与 `reasons` / `items[].recommendReason`（理由）。

### 6.4 配置分层建议

- **推荐侧**（`recommend.*`）：召回开关与各路上限、精排权重（`ranking.*`）、Item-CF 开关与历史深度、`smart-max-pool`、排除分类、冷启动分类、关键词长度、语义/跨域开关与加分上限、AI 头部重排条数 N、反馈埋点开关等。
- **AI 侧**（`ai.recommend.*`）：enabled、provider、api-url、api-key、model、**timeout-ms**（建议 8～20s，超时静默降级）、hot-topics 等。

超时由 **RestTemplate** `aiRecommendRestTemplate` 与 `ai.recommend.timeout-ms` 对齐，避免拖垮 Tomcat 线程池。

### 6.5 可选扩展（答辩亮点）

- **推荐理由缓存**：建议缓存键纳入 **userId**、**有序候选 id**、**进 Prompt 的上下文摘要**（关键词、偏好类目名等）、**模型/Prompt 版本**，以及各候选的 **`updateTime`**（或等价内容版本），避免正文已更新仍复用旧理由；TTL 60～300s；**未登录用户一般不写入个性化理由缓存**（与 6.2 一致）。

## 七、接口约定（逻辑）

| 方法 | 路径 | 说明 | 认证 |
|------|------|------|------|
| POST | /recommend/behavior | 行为上报 | **须登录**（无 userId 则拒绝） |
| GET | /recommend/list | 智能推荐分页；Query：**pageNum、pageSize、keyword、useAi** | **可选**；未登录冷启动，不调用 AI |
| POST | /recommend/feedback | 曝光/点击反馈（`recommend.feedback.enabled`、表见 `DB/17_recommend_feedback_optional.sql`） | **无需登录**（JWT 排除该路径；带 Token 时可写入 userId） |

具体参数名、是否必填以实现代码与 OpenAPI 文档为准。

---

## 八、数据库（行为日志）

核心表 **`user_behavior`**（脚本示例见 `DB/04_user_behavior.sql`）：用户、行为类型、对象类型、对象 id、时间等；宜有 `(user_id, create_time)`、`(target_type, target_id)` 等索引以支撑画像与 Item-CF 查询。

推荐反馈若单独落表，与主行为表分离，便于离线评估 CTR。

---

## 九、实现分层建议

| 层次 | 职责 |
|------|------|
| Controller | 参数校验、限流、解析 userId、调用推荐服务 |
| 推荐服务 | 候选构建、多因子打分、排序、分页、组装 pageHint；在满足条件时调用 AI 服务 |
| AI 服务 | Prompt 拼装、HTTP 调用、解析 order/scores/reasons、失败降级 |
| Mapper / SQL | 行为查询、共现查询、加权分类 TopK 等；SQL 与业务规则保持注释同步 |

业务上 **`recordBehavior`** 应在各模块**统一入口**调用，避免遗漏导致画像偏置。

## 十、扩展与优化方向

- **画像**：引入显式反馈（不感兴趣、负样本）、会话内短期兴趣、分时段衰减。
- **排序**：学习排序（Learning to Rank）或向量召回 + 重排，与现有规则分融合。
- **AI**：RAG 引入规范条文摘要；理由与排序分开展示，避免用户误将理由当作唯一排序依据。
- **工程**：结果短时缓存、画像异步更新、离线评估看板。

## 十一、用户端展示（与当前 client 对齐）

| 项目 | 说明 |
|------|------|
| 页面 | `client/src/views/knowledge/List.vue`：「消防知识」下 **智能推荐 / 按时间 / 热门** 三 Tab；仅 **智能推荐** 调用 `/recommend/list`，后两者调用内容列表接口。 |

## 附录：与「仅分类、无内容标签」的对齐

本模块的用户侧兴趣与内容侧匹配均以**知识分类**为主；不在本文档中定义「知识内容多标签」的画像或排序因子。若产品后续增加维度，应独立成章说明与分类的关系及是否进入 AI context。
