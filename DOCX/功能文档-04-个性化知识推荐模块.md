# 功能文档 - 模块四：个性化知识推荐模块

本文档对应需求与设计中的**个性化知识推荐模块**，说明行为采集、兴趣分析、**推荐计算策略**以及**智能推荐 - AI 模型接入**的代码设计与实现。

---

## 一、功能概述

记录用户浏览知识、点赞帖子、评论、收藏等行为到用户行为表；基于行为做兴趣分析（如最常访问分类 Top1）。推荐计算**有行为时**优先采用**基于物品的协同过滤（Item-CF）**：与用户看过/收藏过的内容“相似”的内容（被同一批用户行为过）按共现得分排序推荐；若未启用或 Item-CF 结果不足则回退为“兴趣分类 Top1 + 高浏览量”规则推荐；**无行为时**冷启动推荐全站热门（浏览量+点赞综合排序），分页返回。

**智能推荐核心 = AI 参与选品 + 排序**：当启用 AI 时，先按上述策略**召回更多候选**（条数由 `recommend.ai.recall-size` 配置），再调用 AI 服务对候选进行**打分或重排序**（reorderByAi），得到“AI 排序后的列表”后再分页；最后对当前页补充推荐理由（recommendReason）。即 AI 不仅生成文案，更参与**推荐顺序的决策**。未配置或调用失败时自动回退为“仅规则/Item-CF + 分页”，不调用 AI。

## 二、需求与设计要点

| 功能点 | 需求描述 | 设计实现要点 |
|--------|----------|--------------|
| 行为采集 | 浏览、点赞、评论、收藏等写入用户行为表 | user_behavior 表：user_id, behavior_type(VIEW/LIKE/COMMENT/COLLECT), target_type(CONTENT/POST), target_id；可由前端显式调用或由内容详情/收藏/论坛等接口内部调用 |
| 兴趣分析 | 用户最常访问分类 Top1、互动最多标签 | 通过 SQL 聚合 user_behavior 及相关表；推荐计算时可用简化逻辑：有行为则按“高兴趣分类+高浏览量”查知识 |
| 推荐计算 | 有行为→Item-CF 或兴趣分类+热度；无行为→全站热门 | 有行为且启用 Item-CF：与用户历史内容共现的其他内容按得分排序；否则按兴趣分类 Top1+view_count；无行为：按 view_count、like_count 排序分页 |
| **智能推荐 - AI 选品+排序** | AI 参与推荐顺序决策：先召回候选，AI 打分/重排后再分页 | 启用时：getRecallCandidates(recall-size) → reorderByAi（通用 API 返回 order/scores，豆包用 prompt 输出 id 顺序）→ 分页 → enrichRecommendReasons 补理由；未启用或失败则沿用规则分页 |
| 结果返回 | 分页返回推荐列表 | PageResult&lt;Map&gt;，list 为知识摘要（id、title、categoryName、cover、summary、viewCount、likeCount、createTime；**启用 AI 时可能含 recommendReason**） |

## 三、基于物品的协同过滤（Item-CF）

### 3.1 思路

**相似度定义**：两个知识内容“相似”当且仅当有**同一用户**对它们都产生过行为（浏览、收藏等）。相似度得分 = 同时行为过这两个内容的**用户数**（共现次数）。

**推荐流程**：取当前用户最近行为过的若干条内容 → 对每条内容查“相似内容”及共现得分 → 聚合去重、累加得分 → 排除用户已看过的 → 按得分降序得到推荐列表，分页返回。

### 3.2 配置（application.yml）

| 配置项 | 类型 | 说明 |
|--------|------|------|
| recommend.use-item-cf | boolean | 是否启用 Item-CF，默认 true；false 时仅用“兴趣分类 Top1 + 热度” |
| recommend.item-cf.user-history-limit | int | 取用户最近行为过的内容数量，默认 30 |
| recommend.item-cf.similar-per-item | int | 每个内容取多少相似内容参与聚合，默认 20 |
| recommend.ai.recall-size | int | 启用 AI 时召回候选条数（供 AI 重排后分页），默认 50 |

### 3.3 实现要点

- **UserBehaviorMapper.findSimilarContentIds(contentId, limit)**：SQL 自关联 `user_behavior`，查与 `contentId` 被同一 `user_id` 行为过的其他 `target_id`（content_id），按共现用户数 `COUNT(DISTINCT user_id)` 降序，返回 contentId 与 score。
- **RecommendServiceImpl.recommendByItemCF**：取用户最近 N 条行为对应的 contentId → 对每个 contentId 调 `findSimilarContentIds` → 在 Java 中聚合（按 contentId 累加 score）、排除用户已看、按 score 排序 → 分页取 contentId 列表 → 查 `knowledge_content` 组装结果；若候选为空或不足则返回 null，上层回退到规则推荐。

## 五、智能推荐 - AI 模型接入

### 5.1 配置说明（application.yml）

| 配置项 | 类型 | 说明 |
|--------|------|------|
| ai.recommend.enabled | boolean | 是否启用 AI 推荐（选品+排序+理由），默认 false |
| ai.recommend.provider | string | 填 `doubao` 时使用豆包/火山方舟等 OpenAI 兼容 chat 接口；不填则使用通用契约 |
| ai.recommend.api-url | string | 豆包：`https://ark.cn-beijing.volces.com/api/v3/chat/completions`；通用模式为自建服务地址 |
| ai.recommend.api-key | string | 豆包/火山方舟在控制台获取的 API Key；通用模式可选 |
| ai.recommend.model | string | 豆包模式下必填：火山方舟的**推理接入点 ID**（如 ep-20241101xxxx） |
| ai.recommend.timeout-ms | int | 调用超时（毫秒），默认 5000 |
| **recommend.ai.recall-size** | int | 启用 AI 时召回候选条数（供 AI 重排后分页），默认 50 |

### 5.2 豆包/国内大模型接入（推荐）

**适用**：豆包（火山方舟）、通义千问、文心一言等提供 **OpenAI 兼容 chat/completions** 的国内大模型。

**步骤一：获取 API 密钥与模型 ID**

- **豆包（火山方舟）**：登录 [火山引擎控制台](https://console.volcengine.com/) → 方舟大模型 → 推理接入点 → 创建接入点，获得 **API Key** 与 **推理接入点 ID**（即 `model`，形如 `ep-20241101xxxx`）。  
- 请求地址（华北2）：`https://ark.cn-beijing.volces.com/api/v3/chat/completions`。

**步骤二：配置 application.yml 或 application-dev.yml**

```yaml
ai:
  recommend:
    enabled: true
    provider: doubao
    api-url: https://ark.cn-beijing.volces.com/api/v3/chat/completions
    api-key: 你的火山方舟_API_Key
    model: ep-xxxxxxxx   # 替换为你的推理接入点 ID
    timeout-ms: 5000
```

**步骤三：行为说明（选品+排序 + 理由）**

- **排序（reorderByAi）**：后端将召回得到的候选列表（id、标题、摘要）拼成 prompt，请求大模型“按对用户的推荐优先级排序，仅输出 id 列表，英文逗号分隔（如 3,1,5,2）”。解析返回的 id 序列后重排候选列表，实现 **AI 参与排序**。  
- **理由（enrichRecommendReasons）**：对当前页列表再请求大模型“为每条生成一句推荐理由（不超过30字），每行一条”。请求体为 OpenAI 格式；响应解析后写入 `recommendReason`。  
- 超时或异常时静默降级：排序失败则使用原顺序，理由失败则不填充 recommendReason。

### 5.3 接口契约（通用模式：自建/第三方服务）

- **请求**：POST，Content-Type: application/json  
  Body：`{ "userId": 用户ID或null, "candidates": [ {"id":1,"title":"...","summary":"...","categoryName":"..."} ] }`
- **响应**（任选其一或同时支持）：  
  - **排序**：`{ "order": [ 2, 1, 3, ... ] }` 表示推荐顺序的 id 列表；或 `{ "scores": [ 0.9, 0.2, ... ] }` 与 candidates 一一对应，后端按 score 降序重排。  
  - **理由**：`{ "reasons": ["理由1","理由2",...] }`，或 `{ "items": [ {"id":1,"recommendReason":"理由1"}, ... ] }`。  

后端先按 order/scores 重排（reorderByAi），再对当前页补 reasons（enrichRecommendReasons）。调用超时或异常时不修改列表。

### 5.4 包与类

| 类 | 说明 |
|----|------|
| UserBehaviorMapper | findSimilarContentIds(contentId, limit)：Item-CF 相似内容查询 |
| AiRecommendService | 接口：**reorderByAi(userId, candidates)** 返回重排列表；**enrichRecommendReasons(userId, items)** 补理由；**isAiRecommendEnabled()** |
| AiRecommendServiceImpl | 排序：豆包用 prompt 输出 id 顺序并重排；通用 API 解析 order 或 scores 并重排。理由：豆包/通用与原先一致 |
| RecommendServiceImpl | 启用 AI 时：getRecallCandidates(recall-size) → reorderByAi → 分页 → enrichRecommendReasons；未启用时沿用 Item-CF/规则分页后再 enrichRecommendReasons |

## 七、代码设计与实现

### 7.1 包与类结构

| 层次 | 类/接口 | 说明 |
|------|---------|------|
| Controller | RecommendController | behavior 上报、recommend/list 分页 |
| Service | RecommendService、RecommendServiceImpl | 依赖 UserBehaviorMapper、KnowledgeContentMapper、KnowledgeCategoryMapper、**AiRecommendService** |
| Service | **AiRecommendService、AiRecommendServiceImpl** | 智能推荐：调用外部 AI 接口为推荐列表补充 recommendReason 或重排序 |
| Entity | UserBehavior | 与表一致 |
| Mapper | UserBehaviorMapper | BaseMapper；**findSimilarContentIds(contentId, limit)** 用于 Item-CF 相似内容查询 |

### 7.2 行为上报实现

- **recordBehavior(userId, behaviorType, targetType, targetId)**：new UserBehavior，set 各字段，userBehaviorMapper.insert。前端在进入内容详情、收藏、点赞、评论等时机调用，或由各业务接口内部调用以统一采集。

### 7.3 推荐列表实现

- **recommendList(userId, pageNum, pageSize)**：  
  - **启用 AI 时（智能推荐核心 = 选品+排序）**：先调用 **getRecallCandidates(userId, hasBehavior, aiRecallSize)** 召回最多 recall-size 条候选（有行为时优先 Item-CF 的 sortedIds 前 N 条，否则兴趣分类+热度或全站热门 limit N）；再调用 **aiRecommendService.reorderByAi(userId, candidates)** 得到 AI 重排后的列表；按 pageNum、pageSize 截取当前页；对当前页调用 **enrichRecommendReasons** 补推荐理由；total 为重排后列表长度，封装 PageResult 返回。  
  - **未启用 AI 时**：有行为且启用 Item-CF 则 recommendByItemCF 分页；否则按兴趣分类+热度或全站热门分页查 DB；将当页结果调用 enrichRecommendReasons（可选补理由），返回 PageResult。  
  - **getRecallCandidates**：内部复用 getItemCFSortedIds（有行为+Item-CF）或按分类/全站 limit recallSize 查询，统一返回 List&lt;Map&gt;。

### 7.4 与内容/论坛模块联动

- 内容详情接口中可增加：view_count 自增、调用 recommendService.recordBehavior(userId, "VIEW", "CONTENT", contentId)。  
- 收藏、论坛点赞/评论时也可上报 COLLECT、LIKE、COMMENT，便于后续兴趣分析细化。

### 7.5 智能推荐 AI 接入实现

- **reorderByAi**：若未启用或候选为空则返回原列表。豆包模式：构造“按推荐优先级排序，仅输出 id 逗号分隔”的 prompt，调用 chat 接口，解析返回的 id 序列后重排列表；通用模式：POST userId+candidates，解析响应 **order**（id 列表）或 **scores**（与 candidates 一一对应，按 score 降序重排），返回重排后的新列表。异常时返回原列表。  
- **enrichRecommendReasons**：与原先一致，将 items 转为 candidates 与 userId 一并 POST（或豆包用“为每条生成一句推荐理由”的 prompt），解析 reasons 写入 recommendReason；若响应含 order 也可重排（通用模式）。异常时静默跳过。  
- **isAiRecommendEnabled**：返回配置 ai.recommend.enabled，供 RecommendServiceImpl 决定是否走“召回→AI 重排→分页”流程。

## 八、扩展建议

- 兴趣分析：用 SQL 统计 user_behavior 中 target_type=CONTENT 时关联的 category_id 出现次数，取 Top1 作为偏好分类。  
- 标签维度：若知识带标签，可统计用户行为涉及的知识标签频次，推荐同标签高浏览内容。  
- 缓存：推荐结果可按 userId 或“热门”做短时缓存，减轻数据库压力。
