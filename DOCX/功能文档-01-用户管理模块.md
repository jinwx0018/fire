   # 功能文档 - 模块一：用户管理模块

本文档对应《毕业设计需求文档》《基于SpringBoot的消防科普推荐系统的设计与实现》中的**用户管理模块**，说明需求、接口、数据库与**代码设计与实现**。

---

## 一、功能概述

用户管理模块负责用户注册、登录、个人信息维护、密码找回、作者申请审核、**站内通知**、**用户自助注销**与账号状态管理。登录采用 JWT + Refresh Token 会话治理，密码使用 BCrypt 加密存储，并记录关键审计日志；管理端支持审计日志分页查询。

---

## 二、需求与设计要点（来源：需求文档 + 设计与实现）

| 功能点 | 需求描述 | 设计实现要点 |
|--------|----------|--------------|
| 用户注册 | 前端校验用户名、手机号格式与查重；密码加密入库 | 后端再次校验用户名/手机号唯一性；使用 BCrypt 加密密码；默认角色为普通用户(role_id=1) |
| 用户登录 | 用户名、密码校验，生成 Token 返回前端 | 校验用户存在、密码匹配、账号状态；签发 accessToken + refreshToken；记录登录审计日志 |
| 信息管理 | 用户可修改头像、手机号、邮箱；参数传后端校验后更新数据库 | 仅允许修改当前登录用户；手机号、邮箱修改时校验不与他人重复；使用 MyBatis-Plus 按 ID 更新 |
| JWT 会话治理 | 避免长期有效 token 风险 | 提供 /user/refresh-token 刷新令牌、/user/logoutAll 退出全部设备；支持管理员强制下线；改密后旧 token 失效（token_version） |
| 密码找回 | 支持邮箱链接找回 | 接口：sendResetEmail、resetByToken；重置链接 30 分钟有效；发送频控（60 秒内不可重复、1 小时最多 5 次）；一次性 token |
| 作者权限申请与审核 | 普通用户可申请成为作者，由管理员审核通过后获得 AUTHOR 角色 | 申请支持申请原因与附件；审核支持备注与驳回原因；审核结果通过站内通知/邮件告知申请人 |
| 账号状态管理 | 管理员可冻结/解冻/注销账号 | 冻结后禁止登录；注销为软删除（deleted=1）；冻结/注销/强制下线均会使旧会话失效 |
| 审计追踪 | 关键操作可追责 | 记录注册、登录、改密、找回密码、角色变更、审核操作、管理端用户操作等审计日志 |

## 四、数据库设计

脚本位置：`DB/01_sys_user.sql`、`DB/07_author_application.sql`、`DB/10_user_security_and_audit.sql`。

| 表名 | 说明 |
|------|------|
| sys_role | 角色表，角色编码 USER/AUTHOR/ADMIN；**固定 id：1=USER，2=AUTHOR，3=ADMIN**（作者申请通过后 `sys_user.role_id` 置为作者角色 id；管理员账号应为 3） |
| sys_user | 用户表，含用户名、密码(BCrypt)、手机、邮箱、头像、角色、状态、token_version、deleted、deleted_time |
| sys_password_reset_token | 邮箱找回密码用 Token（过期时间、是否已使用） |
| user_session | 刷新令牌会话表（refresh token、撤销状态、过期时间） |
| author_application | 作者申请表（申请原因、附件、审核备注、驳回原因） |
| user_notification | 站内通知（作者申请审核结果通知） |
| audit_log | 审计日志（关键行为追踪） |

**sys_user 关键字段：** id, username(唯一), password(128 位存 BCrypt), phone(唯一), email(唯一), avatar, role_id, status, token_version, deleted, deleted_time, create_time, update_time。

### 5.2 认证流程设计

1. **放行路径**：WebMvcConfig 中放行 `/user/register`、`/user/login`、`/admin/login`、`/user/refresh-token`、`/user/password/*` 等。
2. **需认证请求**：请求头携带 `Authorization: Bearer <accessToken>`，拦截器校验 token 有效、type=access、tokenVersion 与数据库一致、账号状态正常。
3. **Token 签发**：登录成功后同时签发 accessToken 与 refreshToken；refreshToken 写入 `user_session`。
4. **Token 刷新**：`/user/refresh-token` 校验 refreshToken、会话有效性与 tokenVersion，成功后返回新 accessToken + refreshToken（旧 refresh 失效）。

### 5.3 注册实现要点

- **UserServiceImpl.register(RegisterReq)**：  
  - 使用 LambdaQueryWrapper 按 username、phone 分别查询，若已存在则抛出 BusinessException("用户名已存在") 或 "手机号已注册"。  
  - 新建 SysUser，password 使用 `passwordEncoder.encode(req.getPassword())` 写入。  
  - roleId 设为 1（普通用户），status 设为 1，tokenVersion=0，deleted=0，执行 userMapper.insert(user)，返回 user.getId()。

### 5.4 登录实现要点

- **UserServiceImpl.login(LoginReq)**：  
  - 校验用户名密码、账号状态（禁用/注销）。  
  - 根据角色生成 accessToken（含 role、tokenVersion）与 refreshToken（type=refresh）。  
  - 持久化 refreshToken 到 user_session。  
  - 返回 LoginVO（token、refreshToken、userId、username、role、avatar）。

### 5.5 信息管理实现要点

- **getInfo(userId)**：根据 userId 查 SysUser，若为空抛 NOT_FOUND；复制到 UserInfoVO，并查角色得到 roleCode 填入。  
- **updateInfo(userId, avatar, phone, email)**：查当前用户；若传入 phone，先查是否被其他用户占用（排除当前 userId），邮箱同理；有则抛 "手机号已被使用" 等；最后 set 字段并 userMapper.updateById(user)。

### 5.6 密码与会话安全实现要点

- **updatePassword / resetPasswordByToken**：密码更新成功后 `tokenVersion + 1`，并将该用户所有未撤销 refreshToken 标记 revoked，确保旧会话全部失效。  
- **sendResetEmail**：增加频控（60 秒内限制重复发送、1 小时最多 5 次），并将历史未使用 token 置为已使用（一次性）。  
- **JwtInterceptor**：仅接受 accessToken；校验 tokenVersion 与账号状态，防止旧 token 越权继续使用。

### 5.7 作者申请与审核实现要点

- **apply(userId, applyReason, attachments)**：普通用户提交申请，记录申请原因与附件；若已有待审核记录则拒绝重复提交。  
- **approve/reject**：管理员审核时记录 reviewRemark/rejectReason，审核通过将 role 调整为 AUTHOR。  
- **通知能力**：审核完成后写入 user_notification；若邮箱已配置，发送审核结果邮件。

### 5.8 管理员用户管理实现要点

- **adminUserList(pageNum, pageSize, username, role)**：使用 MyBatis-Plus 的 Page 与 LambdaQueryWrapper，按 username 模糊、role（需关联 sys_role 或 role_id）筛选，selectPage 后将每条 SysUser 转为 UserInfoVO（含角色码），返回 IPage&lt;UserInfoVO&gt;（JSON 序列化字段为 **records**、total、current、size 等）。
- **adminGetUser / adminCreateUser / adminUpdateUser / adminDeleteUser**：管理端 CRUD；`AdminUserController` 中冻结、解冻通过调用 **adminUpdateUser** 更新 status 实现。
- **冻结/解冻/强制下线/注销**：冻结将 status 置 0；强制下线与注销会触发 tokenVersion 增加并撤销会话；注销采用软删除（deleted=1）。
- **审计**：新增/修改/软删/强制下线会写入 `audit_log`（action 如 `ADMIN_CREATE_USER`、`ADMIN_UPDATE_USER` 等）。**冻结/解冻**走修改用户逻辑，审计记为 **`ADMIN_UPDATE_USER`**，无单独 FREEZE/UNFREEZE 动作码（若需区分可在后续版本扩展）。

### 5.9 站内通知实现要点

- **UserNotificationService**：按 userId 分页查询 `user_notification`，支持 isRead 筛选；**getDetail(userId, id)** 返回单条 Map（与列表字段一致），非本人返回 404；`unreadCount` 统计未读条数；`markRead`/`markAllRead` 更新已读状态。  
- **UserNotificationController**：路径前缀 `/user/notifications`，需登录，从 JWT 中解析 userId；用户端列表可摘要展示，**详情页**请求 `GET /user/notifications/detail/{id}` 阅读完整正文。

### 5.10 用户自助注销实现要点

- **UserServiceImpl.selfDeleteAccount**：校验密码；若角色为 ADMIN 则拒绝；软删除、提升 tokenVersion、撤销 refresh 会话。  
- 与管理员在后台「删除用户」均为软删除，区别在于管理员可操作任意非自身账号，用户端仅可操作本人且需密码确认。

### 5.11 作者申请附件与头像上传

- **uploadAuthorAttachment**：`multipart/form-data`，白名单扩展名与 5MB 限制，保存至 `uploads/author_attachments/`，返回可访问 URL。  
- **uploadAvatar**：上传至 `uploads/avatar/`（或配置目录），返回 URL 供前端写入用户信息。

### 5.12 管理端审计日志

- **AuditLogServiceImpl.pageForAdmin**：按 action、operatorId 可选筛选，分页返回 `audit_log` 表记录，供管理端「审计日志」页面展示。

## 六、补充说明

- 若未配置邮件服务，系统仍可生成重置链接（输出到日志）用于开发联调。  
- 管理端必须使用 `/admin/login` 登录；普通用户与作者账号不可登录管理端。  
- 老库升级请执行 `DB/10_user_security_and_audit.sql`，以补齐会话治理、审计与通知相关表结构。  
- 分页接口若返回 `records` 字段，为 MyBatis-Plus `IPage` 序列化结果，与部分文档中的 `list` 表述等价，前端按实际字段解析即可。  
- 管理端登录与普通用户登录共用 **login** 流程，成功审计记 **`USER_LOGIN`**（未单独拆分 ADMIN_LOGIN 动作码）。

### 6.1 匿名接口限流（新增）

- **实现类**：`PublicEndpointRateLimiter` + `SlidingWindowCounter`（单机内存窗口）。  
- **覆盖**：`POST /user/register`（按 IP / 小时）、`POST /user/login` 与 `POST /admin/login`（按 IP、用户名 / 分钟）、`POST /user/password/sendEmail`（在原有邮箱维度限频基础上增加按 IP / 小时）。  
- **配置**：`application.yml` 中 `app.rate-limit.*`，详见《后端接口文档》**§1.5**。

### 6.2 末位管理员保护（新增）

- **adminDeleteUser**：禁止删除**当前登录管理员本人**；禁止删除**最后一个可登录管理员**（角色 ADMIN + status=1 + 未注销）。  
- **adminUpdateUser / freeze**：若将导致某用户从「可登录管理员」变为非管理员或不可用，且系统中仅剩该一名可登录管理员，则拒绝并提示业务错误。  
- **实现**：`UserServiceImpl` 中 `isActiveAdmin`、`countActiveAdmins`、`simulateUserAfterAdminUpdate`、`assertNotRemovingLastActiveAdmin`；管理端接口传入 `operatorId`（从 JWT 解析）。

### 6.3 接口文档与 JWT 排除路径（新增）

- **文档访问**：`JwtInterceptor` 根据 `app.api-docs.enabled`、`app.api-docs.restrict-to-internal-network` 对 `/doc.html`、`/v3/api-docs`、`/webjars/**` 返回 404 或 403；生产见 `application-prod.yml`。  
- **JWT**：`WebMvcConfig` 支持 `jwt.extra-exclude-paths` 追加无需 Token 的路径。
