## 二、系统概述

### 2.1 项目背景与目标

本系统为消防科普推荐系统，面向普通用户与管理员，提供消防知识浏览、论坛交流、器材查询、新闻查看及个性化推荐等功能，旨在提升公众消防意识与自救能力。个性化推荐模块支持在规则推荐基础上可选接入 AI 模型，对推荐结果进行增强（补充推荐理由或重排序），提升智能推荐的可解释性与精准度。

### 2.2 用户角色

| 角色 | 说明 | 主要能力 |
|------|------|----------|
| 普通用户 | 注册登录后的访客 | 浏览知识、收藏/分享、发帖评论点赞、查看推荐、器材查询、查看新闻，可发起作者权限申请 |
| 作者 | 具备内容发布权限的用户 | 在普通用户能力基础上，可发布/管理消防知识内容，但发布内容需经过管理员审核通过后才对外展示 |
| 管理员 | 后台管理人员 | 用户/内容/帖子/器材/新闻管理、审核（含作者权限申请与知识内容发布审核）、**数据统计（含全站运营大盘、区间内按月趋势）**与导出 |

### 2.3 技术选型（建议）

| 层次 | 技术 | 备注 |
|------|------|------|
| 后端框架 | Spring Boot | 主框架 |
| 安全与认证 | JWT | 登录态、Token 返回 |
| 数据库 | MySQL 5.6 | 业务数据（建表脚本见 DB 文件夹，兼容 5.6.x） |
| 缓存 | Redis | 点赞 Set、会话等 |
| 分页 | 自定义或 MyBatis-Plus / HelpPage | 分页查询 |
| 邮件 | JavaMail（Spring Mail） | 密码找回（邮箱重置链接） |
| 报表导出 | EasyExcel | 统计结果导出 Excel |
| 智能推荐 - AI 模型 | 可选豆包（OpenAI 兼容）或通用 HTTP | 服务端 `ai.recommend.*`；列表需 **`useAi=true`** 且用户已登录才调用大模型；用户端默认不勾选「AI 增强」 |
| 前端 | Vue3 | admin + client 两套前端 |

## 三、功能需求详述

### 模块一：用户管理模块

#### 3.1.1 用户注册

| 需求项 | 描述 |
|--------|------|
| 功能简述 | 新用户通过前端提交注册信息，后端校验并落库。 |
| 前端校验 | 校验用户名格式、手机号格式；与已有数据查重（如手机号唯一）。 |
| 密码处理 | 使用加密算法（如 BCrypt）对密码加密后存入数据库，禁止明文存储。 |
| 入库内容 | 至少包含：用户名、手机号、加密密码、注册时间等；可选邮箱、角色。 |
| 异常处理 | 手机号/用户名重复、格式错误时返回明确错误信息。 |

#### 3.1.2 用户登录

| 需求项 | 描述 |
|--------|------|
| 功能简述 | 用户名 + 密码校验，通过后颁发 JWT Token。 |
| 输入 | 用户名、密码。 |
| 校验 | 校验用户存在性、密码与库中密文一致（需用相同加密算法比对）。 |
| 输出 | 成功时返回 accessToken、refreshToken 及必要用户信息（如 userId、角色）；失败返回错误码与提示。 |
| 安全 | accessToken 短期有效、refreshToken 用于续期；冻结/改密/注销后旧 token 失效。 |

#### 3.1.3 信息管理

| 需求项 | 描述 |
|--------|------|
| 功能简述 | 用户可修改个人资料：头像、手机号、邮箱等。 |
| 流程 | 前端将修改项传给后端；后端做格式与唯一性校验（如手机号、邮箱不与他人重复）。 |
| 持久化 | 使用 SQL（或 ORM 更新）更新对应用户记录。 |
| 权限 | 仅允许修改当前登录用户自身信息；敏感操作可要求再次验证。 |

#### 3.1.4 密码找回

| 需求项 | 描述 |
|--------|------|
| 方式 | 邮箱链接找回：使用 JavaMail（Spring Mail）发送带重置链接的邮件，链接有效期内可重置密码。 |
| 安全 | 链接有效期、次数限制、重置后旧 Token 失效等需在实现时考虑。 |

#### 3.1.5 作者权限申请与审核

| 需求项 | 描述 |
|--------|------|
| 功能简述 | 普通用户可在个人中心或导航中发起“申请成为作者”，管理员在管理端审核通过后，将其角色从普通用户变更为作者。 |
| 申请流程 | 用户端提交作者申请；系统记录到作者申请表（含用户 ID、状态、审核人、审核时间、驳回原因）；同一用户在上一条申请待审核时不可重复提交。 |
| 审核流程 | 管理端提供作者申请列表页，列出待审核申请；管理员可对单条申请执行“通过”或“驳回”，通过后为用户赋予 AUTHOR 角色，驳回时填写驳回原因，用户可在申请状态页查看。 |
| 约束 | 仅未成为作者的普通用户可申请；一条申请处理完毕前不可重复发起。 |

#### 3.1.6 会话与安全（双令牌）

| 需求项 | 描述 |
|--------|------|
| 功能简述 | 登录后颁发 accessToken 与 refreshToken；支持刷新令牌、退出全部设备、管理员强制下线；冻结/改密/注销后旧 token 失效。 |
| 实现要点 | 用户会话与 tokenVersion 校验；refreshToken 可持久化至 `user_session` 表。 |

#### 3.1.7 站内通知与审计

| 需求项 | 描述 |
|--------|------|
| 站内通知 | 登录用户可分页查看站内消息、未读数量，支持单条或全部标记已读；可通过**单条详情接口**查看完整正文（如作者申请审核结果等场景）。 |
| 审计日志 | 关键操作记录至 `audit_log`；管理员可在后台分页查询（按操作类型、操作人筛选）。 |

#### 3.1.8 用户自助注销

| 需求项 | 描述 |
|--------|------|
| 功能简述 | 用户在个人中心可发起账号注销，需输入登录密码确认。 |
| 约束 | 采用软删除；**管理员账号不可在用户端自助注销**；注销后会话失效。 |

### 模块二：消防知识内容管理模块

#### 3.2.1 内容发布与作者审核流程

| 需求项 | 描述 |
|--------|------|
| 功能简述 | 管理员或作者可发布/更新消防知识内容并保存到数据库。作者发布时先保存为草稿，提交审核后由管理员审核通过才对前台用户展示。 |
| 内容字段 | 至少包含：标题、分类、正文、封面/图片、作者、发布时间、状态（0 草稿，1 已发布，2 下架，3 待审核）、最近一次驳回原因等。 |
| 权限与流程 | 仅管理员、作者角色可创建内容。作者创建内容后初始为草稿，可多次编辑；点击“提交审核”后状态变为待审核(3)，管理员在管理端对待审核内容执行通过(1)/驳回(0)；驳回时需填写驳回原因并记录，作者可在“我的内容”列表与详情中查看并根据原因修改后重新提交。 |

#### 3.2.2 内容查询

| 需求项 | 描述 |
|--------|------|
| 功能简述 | 支持按分类、标题、标签筛选，并分页查询。 |
| 实现建议 | 使用 HelpPage 或 MyBatis-Plus 等实现分页；条件：分类 ID、标题关键词、标签；支持排序参数（最新/热度）。 |
| 返回 | 分页列表（当前页、每页条数、总数、列表数据）。 |

#### 3.2.3 内容管理（增删改查）

| 需求项 | 描述 |
|--------|------|
| 功能简述 | 作者、管理员对知识内容进行增、删、改、查。 |
| 权限 | 作者可管理自己发布的内容；管理员可管理全部。删除采用逻辑删除并进入回收站。 |

#### 3.2.4 内容状态与回收站

| 需求项 | 描述 |
|--------|------|
| 发布状态 | 内容状态包含草稿、待审核、已发布、下架；管理员可执行“下架/恢复发布”。 |
| 回收站 | 已逻辑删除内容可在管理端回收站分页查看并恢复。 |
| 审计 | 删除、恢复、下架、恢复发布等关键内容操作需记录审计日志，支持管理端追溯。 |

#### 3.2.5 内容收藏、分享与点赞

| 需求项 | 描述 |
|--------|------|
| 收藏 | 用户可收藏某条知识内容；需新增 **user_collection**（用户收藏表），关联用户 ID、内容 ID、收藏时间。 |
| 分享 | 支持生成分享链接（如带 contentId 的 H5 链接），便于分享给他人查看。 |
| 去重 | 同一用户对同一内容仅记录一条收藏。 |
| 点赞 | 用户可点赞/取消点赞；内容作者可收到站内通知；作者在“我的内容”可查看点赞数。 |

### 模块四：个性化知识推荐模块

#### 3.4.1 行为采集

| 需求项 | 描述 |
|--------|------|
| 功能简述 | 记录用户各类行为，写入用户行为表。 |
| 行为类型 | 浏览知识、点赞帖子、评论帖子、收藏内容等。 |
| 表设计 | 用户行为表建议包含：用户 ID、行为类型、关联 ID（知识 ID/帖子 ID）、时间等，便于后续统计与推荐。 |

#### 3.4.2 兴趣分析

| 需求项 | 描述 |
|--------|------|
| 功能简述 | 基于用户行为表构建**分类维度**兴趣画像（隐式反馈 + 时间衰减）。 |
| 统计内容 | 用户对知识内容的偏好分类序列（如 Top3）；与全站运营统计区分。 |
| 实现说明 | 当前应用以 **knowledge_category** 关联画像；**内容多标签**表若存在亦**不参与**推荐列表与排序（与数据导入说明一致）。 |

#### 3.4.3 推荐计算策略

| 策略 | 描述 |
|------|------|
| 数据范围 | 仅 **已上架** 消防知识；个人中心「不感兴趣分类」与运营配置排除分类在**候选筛选阶段**过滤（仅智能推荐列表；「按时间」「热门」不受影响）。 |
| 已登录、有个人行为 | **召回**：合并偏好类目较新内容、Item-CF（基于物品的协同）候选、热度暖池等（`recommend.recall.*`）。**精排**：分类匹配阶梯加分、Item-CF 分项（设上限防单因子碾压）、浏览/点赞对数热度、发布时间衰减、可选轻量文本相似（近期浏览 vs 候选标题摘要）、可选论坛标题跨域加分（`recommend.ranking.*` 等）。 |
| 未登录 | **冷启动**：不使用个人画像与 Item-CF/语义/论坛加成；不调用 AI。 |
| 已登录、行为极少 | 偏冷启动 + 全站热度/时效，随行为积累逐步个性化。 |
| 结果 | 分页返回推荐列表；可返回 **pageHint**（策略说明）、**recommendSourceLabel** / **sortExplain**（可解释性）、可选 **recommendReason**（见 3.4.5）。 |

#### 3.4.4 结果返回与前端展示

| 需求项 | 描述 |
|--------|------|
| 接口 | `GET /recommend/list`：分页参数 **pageNum、pageSize**；可选 **keyword**（标题/摘要模糊）；可选 **useAi**（是否请求 AI 增强，默认 false）。 |
| 认证 | **JWT 可选**：未登录可访问列表（冷启动）；行为上报 `POST /recommend/behavior` 须登录。 |
| 用户端页面 | 「消防知识」下 **智能推荐 / 按时间 / 热门** 三 Tab：智能推荐走本接口；后两者走内容列表接口。智能推荐 Tab 提供 **课堂演示模式**（记录 Top5、注入浏览/点赞/收藏后对比排名变化，需登录）。 |
| 数据 | 标准分页结构 + 知识条目字段；启用 AI 且成功时单项可含 **recommendReason**。 |

#### 3.4.5 智能推荐 - AI 模型接入（可选）

| 需求项 | 描述 |
|--------|------|
| 功能简述 | 在规则召回与多因子精排之后，**可选**调用大模型对排序**头部**微调并生成短理由。 |
| 触发条件 | 须同时满足：**useAi=true**、用户**已登录**、服务端 **ai.recommend.enabled** 且密钥/模型就绪；否则静默降级为规则序。用户端 **默认不勾选**「AI 增强」，避免误耗 Token。 |
| 增强方式 | 仅对规则序前 **N** 条（`recommend.ai.rerank-pool-size`）做重排；为**当前页**条目生成 **recommendReason**；失败或超时降级。 |
| 配置项 | `ai.recommend.enabled`、`provider`、`api-url`、`api-key`、`model`、`timeout-ms`、可选 `hot-topics` 等（详见 `application.yml` 注释）。 |
| 限流 | 启用 AI 时可对 `GET /recommend/list` 按 IP/用户限流（`app.rate-limit.recommend-list-ai-*`）。 |
| 接口契约 | 豆包类走 OpenAI 兼容 chat；或通用 HTTP 传入 userId、candidates、context，响应 **order** / **reasons**（与实现类一致）。 |
| 反馈埋点 | 可选 `POST /recommend/feedback`（EXPOSE/CLICK），表结构见 `DB/17_recommend_feedback_optional.sql`。 |

### 模块七：数据统计与分析模块

#### 3.7.1 数据统计维度

| 维度 | 统计内容 |
|------|----------|
| 内容维度 | 按分类统计知识发布量；浏览量 Top10 知识。 |
| 用户维度 | 活跃用户数（如近 7 天有行为）；各角色用户数量。 |
| 互动维度 | 帖子评论数、点赞数；可扩展帖子数、评论数趋势等。 |
| 全站运营大盘 | 在可选时间范围内汇总**知识、新闻、上架器材、论坛互动、行为浏览（VIEW）**等核心指标，管理端以卡片形式一屏展示，便于运营总览。 |

#### 3.7.2 可视化展示

| 需求项 | 描述 |
|--------|------|
| 工具 | 使用 **ECharts** 进行图表展示。 |
| 图表类型 | 柱状图、折线图、饼图等，例如：分类知识发布量饼图；**折线图支持按管理端所选日期区间展示按月趋势**（如已发布知识按创建月对浏览数字段汇总、与用户行为表中浏览行为按月次数可双序列对比）；未选区间时可按当年默认展示。 |
| 数据来源 | 由后端提供统计接口，前端调用后渲染 ECharts。 |

#### 3.7.3 数据导出

| 需求项 | 描述 |
|--------|------|
| 功能简述 | 将统计结果导出为 Excel 文件。 |
| 实现建议 | 使用 **EasyExcel** 生成 Excel，后端提供导出接口（如按统计类型、时间范围导出）。 |

## 四、非功能需求（建议）

| 类型 | 说明 |
|------|------|
| 性能 | 列表与推荐接口需分页，避免单次数据量过大；点赞等高频操作依赖 Redis。 |
| 安全 | 密码加密存储、JWT 校验、接口按角色鉴权、防重复提交与基础防刷。 |
| 可用性 | 关键接口有明确错误码与提示，便于前端与联调。 |
| 可维护性 | 模块划分清晰，后端按模块分包；接口风格统一（如 REST），便于编写接口文档。 |

## 五、数据库与存储要点（概要）

- **用户与认证**：用户表、角色表（若多角色）、JWT 黑名单或白名单（若做登出/刷新）。
- **知识内容**：知识表、分类表、标签表、**user_collection** 收藏表。
- **论坛**：帖子表、评论表；点赞用 Redis Set，可选同步到 DB 的计数字段。
- **行为与推荐**：用户行为表；可建分类/标签统计缓存或定时任务。
- **器材**：器材表（含类型、步骤、检查要点、故障处理等）。
- **新闻**：新闻表（含地区、紧急等级等）。
- **审核与状态**：帖子状态（1/0/-1）、驳回理由字段或表。

具体表结构应在《数据库设计说明书》或后端设计中细化。

## 六、接口与项目结构建议

### 6.1 后端（backend）包结构建议

```
backend/
├── controller/     # 按模块划分：user、content、forum、recommend、equipment、news、statistics
├── service/
├── mapper / repository
├── entity / po
├── dto / vo
```

### 6.2 前后端协作

- 管理端（**admin**）调用：用户管理、内容/帖子/器材/新闻的增删改查、审核、统计与导出等接口。
- 用户端（**client**）调用：注册登录、个人中心、知识浏览与收藏分享、论坛发帖评论点赞、推荐列表、器材查询、新闻列表等接口。
- 建议单独维护一份《前后端接口文档》（按模块列出 URL、方法、参数、返回值），便于联调与毕设答辩说明。
